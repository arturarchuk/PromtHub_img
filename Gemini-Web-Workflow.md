# Gemini Web Workflow

Этот файл задает основной рабочий сценарий PromptHub через веб-версию Gemini / Nano Banana 2.

## Главный сценарий

По умолчанию работаем так:

1. Пользователь описывает идею изображения обычным языком.
2. PromptHub пишет готовый prompt в формате:
   - Русский prompt
   - English prompt
3. Пользователь копирует English prompt в веб-Gemini.
4. Пользователь генерирует изображение в Gemini по своей подписке.
5. Пользователь присылает результат обратно в PromptHub.
6. Пользователь дает короткую обратную связь: что нравится, что не нравится, что нужно сохранить и что изменить.
7. PromptHub анализирует результат через обратную связь пользователя, записывает уроки и делает v2/v3.

## Почему web workflow основной

Веб-Gemini использует лимиты и возможности пользовательской подписки.

Gemini API использует отдельные API quotas и billing Google Cloud. Поэтому API не является основным путем, если пользователь хочет работать по своей веб-подписке.

## Как писать следующий этап

Для обычных prompt-ответов следующий этап должен быть таким:

Следующий этап: сгенерируй изображение в веб-Gemini по English prompt, пришли результат и коротко напиши, что нравится/не нравится и что нужно изменить.

Можно адаптировать формулировку под задачу:

- Для портрета: пришли результат и отметь, что по лицу, коже, глазам, рукам и свету нравится или не нравится.
- Для товара: пришли результат и отметь, что по форме, материалу, логотипу, фону и коммерческому виду нужно сохранить или изменить.
- Для постера: пришли результат и отметь, что по тексту, иерархии, читаемости и композиции работает или нет.
- Для editing: пришли результат и отметь, что изменилось правильно, что нельзя было трогать и что нужно поправить.
- Для референсов: пришли результат и отметь, насколько сохранились identity/product/style consistency.

## Когда использовать API

API использовать только если пользователь явно просит:

- запустить локальный API test
- проверить billing/quota
- сгенерировать через API
- автоматизировать batch workflow

Во всех остальных случаях основной путь - веб-Gemini.

## Что просить у пользователя после генерации

Минимум:

- прислать изображение результата
- коротко написать, что нравится
- коротко написать, что не нравится
- указать, что нужно сохранить в следующей версии
- указать, что нужно изменить в следующей версии

Не делай v2 только по изображению, если пользователь еще не дал направление и проблема не очевидна. Сначала попроси короткую обратную связь.

Если задача сложная:

- какая модель/режим использовались в Gemini
- были ли референсы
- какой prompt был отправлен
- что нужно сохранить в v2
- что нужно изменить в v2

## Feedback loop

Если результат плохой или смешанный:

1. Отделить мнение пользователя от собственной визуальной диагностики.
2. Записать короткий урок в `Журнал-экспериментов.md`.
3. Добавить теги из `Memory-Tags.md`.
4. Сделать v2/v3 prompt.

Если результат хороший:

1. Сохранить компактную запись в `Мои-удачные-промты.md`, если prompt/context известен.
2. Добавить теги.
3. Зафиксировать, почему prompt сработал.
